Lessons from std::array

Created Feb 5, 2026 - Last updated: Feb 5, 2026

programming c++

As a learning exercise, I’ve just started re-implementing some of the C++ STL. Maybe I’ll write more generically about it in future, but the basic idea is:

  • Copy the tests from llvm libc++
  • Re-implement interesting/notable parts of the STL

The idea here is that ’library code’ is typically the ‘most C++’ that you can get. It requires a lot of attention to detail to properly satisfy all of the contracts.

Here’s what I learned re-implementing std::array, in list-icle format.

Writing CTAD Deduction Guides

CTAD is Constructor Template Argument Deduction. The basic idea is that normally the compiler can deduce template arguments from the arguments that you construct an object with. But if the type of the argument is not exactly the same type as what you want the argument to be, then you have to help the compiler out. Consider:

template <typename T1, typename T2>
struct pair {
    pair(T1, T2);
    pair(T1); // construct a pair where first and second are the same element
};

(This example is a little contrived. It’s not a good idea to add a pair constructor like this, but humour me) If we construct pair with the first constructor, we don’t need to provide any template arguments; it will deduce them automatically. We aren’t so lucky with the second.

pair(1, 2); // fine -> pair<int, int>
pair(1); // not fine -> what is T2?

The compiler can’t automatically deduce what T2 is in this case, so it rightfully complains. To fix this, we can add a deduction guide which tells the compiler how to handle this particular case:

template <typename T>
pair(T t) -> pair<T, T>

This says “when someone calls the second constructor, deduce the template arguments for pair as pair<T, T> where ‘T’ is the argument type”. The syntax is a little irregular (and so I find it hard to remember), but it’s also quite sensible. I’m slightly surprised that there’s no scoping of ‘pair(T t)’ to declare which class it’s in e.g. pair::pair(T t) but I suppose that because it’s only possible for constructors, that was elided.

In any case, this is relevant for std::array because it needs some deduction guides, in particular:

template <typename T>
array(initializer_list<T> elems) -> array<T, elems.size()>;

template <typename T, typename... Ts>
array(T&& t, Ts&&... ts)
    -> array<T, sizeof...(ts) + 1>;  // TODO: require Ts are same

The first of these makes perfect sense and is a better motivating use-case for CTAD. It’s obvious to the compiler what the array element type should be from initializer_list<T>, but it’s not obvious what the size should be, so we add a deduction guide for it.

The second of these exists to construct array directly from some elements e.g.:

auto a = std::array{1, 2, 3};

This doesn’t invoke the std::initializer_list constructor1 and so we need a guide, again to tell the compiler how many elements we have. I’m not very confident in my implementation here. I split off T to make it easier to specify the type of the array, but this feels error-prone to me. In any case, it passes the tests for now.

CTAD isn’t so new for me, but it was a nice reminder on how to write deduction guides, as it’s not something that one does often in day-to-day usage.

Zero-Element Arrays

I was surprised by how many of the libc++ tests test edge cases around zero-length arrays. I’ve never before actually constructed a zero-length array, although I understand that it might be useful in some generic code. A lot of my time in the implementation was spent trying to handle this case.

First of all, what are the semantics of a C-style 0-length array? Well cppreference says:

expr [expression for the array size] - […], which evaluates to a value greater than zero but also When used with new[]-expression, the size of an array may be zero; such an array has no elements:

and then for std::array in particular:

There is a special case for a zero-length array (N == 0). In that case, array.begin() == array.end(), which is some unique value. The effect of calling front() or back() on a zero-sized array is undefined.

Ok so essentially a C-style 0-element array won’t compile, but we need to support a size of zero for std::array.

The first test issue that I ran into trying to handle zero-length arrays was essentially as below:

struct NoDefault {
    NoDefault(int) {}
};

struct Arr {
    NoDefault arr[0];
};

int main(int argc, char **argv) {
    Arr arr;
    (void)arr;
    return 0;
}

which gives somewhat cryptic error message:

<source>:21:9: error: call to implicitly-deleted default constructor of 'Arr'
   21 |     Arr arr;
      |         ^
<source>:17:15: note: default constructor of 'Arr' is implicitly deleted because field 'arr' has no default constructor
   17 |     NoDefault arr[0];
      |               ^

Weirdly, I haven’t really figured out exactly why this is the error message given. The best that I can tell is that clang and gcc actually allow zero-length C-style arrays as an extension, and so it fails like this instead of complaining about the zero-length array. It’s still weird that given that zero-length arrays would be permitted, that it would even try to construct NoDefault though.

In any case, to fix this I figured it makes sense to use some template specialization for the zero-case in std::array. Something like:

template <typename T, size_t N>
struct array_base {
    T mElems[N];
};

template <typename T>
struct array_base<T, 0> {};

which works fine for the most part2. The main problem with this approach is that we still need switching in some of the operators. For example, to implement operator[] you need to have something like:

constexpr T& operator[](size_type pos) noexcept {
    if constexpr (N == 0) {
        // abort, return ref to nullptr... or something
    } else {
        return mElems[pos];
    }
}

If you implement most of the other methods by using operator[], then the switching doesn’t sprawl too far, but still it’s non-ideal.

So there’s a trick that we can do here such that mElems exists in both the zero and non-zero case and all of the rest of the code works the same for both. The trick is just to make mElems a T* pointer to nullptr in the zero element case. Because of the adopted-from-C interchangeable syntax for dereferencing pointers with array notation, most of the methods work fine in both cases.

Of course, it wouldn’t be C++ if there weren’t a few exceptions. For example, one libcxx test tests that you can call .swap on a zero-length array with an element type that isn’t swappable… In this case a switch is still required to ensure we don’t try to compile a call to .swap in this path.

So the final code becomes something like:

template <typename T, size_t N>
struct array_base {
    T mElems[N];
};

template <typename T>
struct array_base<T, 0> {
    T* mElems{nullptr};
};

template <typename T, size_t N>
struct array final : array_base<T, N> {
    constexpr T& operator[](size_type pos) noexcept {
        return (*this)[pos]; // UB if called in zero-length case
    }

    constexpr void swap(array& other) noexcept {
        if constexpr (N == 0) {
            return;
        } else {
            for (size_type i{}; i < size(); i++) {
                // without the if constexpr above, compilation will fail here in the zero-length case
                // if T is not a swappable type
                ::std::swap((*this)[i], other[i]);
            }
        }
    }
};

Ok this looks pretty ok. But unfortunately there’s still an issue raised by the libc++ tests.

    // const arrays of size 0 should disable the implicit copy assignment operator.
    typedef std::array<double const, 0> Array;
    Array array = {};
    Array copy  = array;
    (void)copy;
    static_assert(std::is_copy_constructible<Array>::value, "");
    static_assert(!std::is_copy_assignable<T>::value, "")

The idea here is that for non-zero length arrays, you can’t copy-assign them if their element type is const. With our trick to use a pointer for zero-length arrays, we’re able to copy-assign them.

As far as I can tell here, the fix is just to explicitly disable the copy-assignment for this case. I’d like to find out if I can do this more simply using concepts3, but for now, something like this works (i.e. we add another specialization):

template <typename T>
struct array_base<const T, 0> {
    const T* mElems{nullptr};

    array_base& operator=(const array_base&) = delete;
};

Phew.

-Wunqualified-std-cast-call ?

I rarely write c++ within the std namespace. For this exercise, everything is in std! Yay for trying new things. A side-effect of this is that I tried something like the following:

template <size_t Pos, typename T, size_t N>
constexpr const T&& get(const array<T, N>&& a) noexcept {
    return move(a[Pos]);
}

and I was very surprised when this yielded an error message

/Users/tim/repos/stl/include/stl/array/array.h:208:12: warning: unqualified call to 'std::move' [-Wunqualified-std-cast-call]
  208 |     return move(a[Pos]);
      |            ^
      |            std::

I’d never seen this before and didn’t know the reasoning for it.

A quick google turns up this change which says:

This adds a diagnostic when an unqualified call is resolved to std::move or std::forward. This follows some C++ committee discussions where some people where concerns that this might be an usual anti pattern particularly britle worth warning about - both because move is a common name and because these functions accept any values. This warns inconditionnally of whether the current context is in std:: or not, as implementations probably want to always qualify these calls too, to avoid triggering adl accidentally.

Now we know.

empty() and max_size()

I’ve never called these methods on a std::array before, so I was surprised to see them listed in the cppreference api listing. As far as I know, nothing in the STL actually relies on these methods existing. These methods are required because they are listed in the Container named requirements.

In case it’s not obvious (it wasn’t that obvious to me):

  • empty returns true only when the array is of zero-length
  • max_size returns the array length

operator== and operator <=>

I was excited to try out the new C++20 three-way-comparison (aka spaceship) operator, and luckily the std::array reference even says that this is how it’s implemented now. I tried it out and was very surprised to see:

tests/libcxx/support/test_comparisons.h|129 col 56| error: invalid operands to binary expression ('const std::array<int, 3>' and 'const std::array<int, 3>')
||   129 |     ASSERT_SAME_TYPE(decltype(std::declval<const T&>() == std::declval<const U&>()), bool);

This rule doesn’t seem very well-publicised (I was unable to find it on cppreference, but perhaps it’s user error), but the blog post here4 provides some insight:

You might wonder why we can’t express the equality operators in terms of the three-way comparison operator as well. The reason for this, like with many other things in C++, is performance: for certain classes, operator== can be implemented in a more performant manner than operator<=>. One example here is container classes: operator== can return early if the sizes of the two containers don’t match, while operator<=> has to compare individual elements until it finds a difference if it performs a lexicographical comparison.

So it seems that it’s done this way to make the user think about it. Given it was quite hard for me to find this reasoning, I’m not sure how well this is working, but at least now I know. I don’t write C++ because it’s intuitive, that’s for sure.

implementing rbegin

From past discussions in my life and my brief usage of reverse iterators I was somewhat aware that they have a slightly quirky implementation. I was curious as to why and so I went with the naive implementation first. Something like:

struct reverse_iterator {
    constexpr T& operator*() noexcept { return *t; }
    constexpr reverse_iterator& operator++() { --t; return *this; }
    constexpr auto operator<=>(const reverse_iterator& a) const noexcept = default;

    T* t{};
};

constexpr reverse_iterator rbegin() noexcept {
    return reverse_iterator{data() + N - 1};
}
constexpr reverse_iterator rend() noexcept {
    return reverse_iterator{data() - 1};
}

The idea here is that this is the naive ‘reverse’ of normal iteration. Here, your begin is at the last element and your end is one past (or in this reverse case it’s before) the first element. The problem? When you try to use this constexpr, you get an error because you’re supplying a pointer to an element before the array, something like:

<source>:36:57: note: cannot refer to element -1 of array of 3 elements in a constant expression
   36 |     constexpr auto rend() { return reverse_iterator{arr - 1}; }

So the solution here is to just store the iterators in essentially the same way as for the forward case, and then subtract one when you’re dereferencing. Of course, at the time I didn’t realise that one needn’t even do this by hand at all, because of the next item:

std::reverse_iterator

I’d never used this before and didn’t know it existed. Initially I started implementing reverse iterator by myself before chancing upon this. The basic idea is that it takes a (bidirectional) forward iterator and adapts it to be a reverse iterator. Handy!

Conclusion

I was glad that this exercise turned out to be as instructive as I was hoping. I’ve been writing C++ professionally for years, and it can be hard to find new useful things to learn at this point. It’s also the sort of thing that would have been much harder to motivate if I didn’t have AI tools to help me wrangle the tests. Admittedly the tests themselves have existed in libc++ for a long time, but integrating them into a new repo and getting them working would have been absolute drudge-work that I’m not sure I would have attempted without (in this case) codex.
